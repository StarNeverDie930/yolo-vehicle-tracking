## 为什么

用户在启动 PySide6 桌面应用、完整处理一次视频并使进度达到 100% 后，程序出现卡死并退出，终端输出 `QThread: Destroyed while thread '' is still running`。代码排查显示，当前 `ProcessingWorker` 使用了与 `QThread.finished` 同名的业务完成信号，并且主窗口在业务完成回调中立即将 `self.worker` 置为 `None`，存在后台线程尚未真正退出就销毁 QThread 对象的风险。

同时，处理完成后的 GUI 状态还不完整：停止按钮、导出按钮、打开视频、热力图查看/保存、视频未导出提示等状态没有形成明确状态机。普通处理完成后系统会保存分析结果，但不会保存处理后视频；如果用户直接打开另一个视频，当前实现无法提示“已有处理完成但未保存的视频”。这会影响毕业设计演示稳定性，也会让用户误以为处理结果已经完整保存。

任务书和开题报告是验收基线，本变更不修改 `docs/任务书.md` 和 `docs/开题报告.md`，只修正桌面应用运行稳定性和交互闭环。

## 改什么

- 重新定义 GUI 后台任务生命周期，区分业务处理完成、用户取消、错误退出和 QThread 真实结束。
- 避免自定义业务信号与 `QThread.finished` 冲突，确保 worker 对象只在线程真正结束后清理。
- 完善处理完成后的控件状态：停止按钮置灰，允许导出视频、打开新视频、查看热力图和保存热力图。
- 增加“处理完成但视频未导出”的状态判断；打开新视频前提示用户确认。
- 导出视频期间进入 busy 状态，禁止重复处理、重复导出、切换视频或查看热力图。
- 停止处理必须区分“用户取消”和“正常完成”，避免取消后误报处理完成。
- 程序关闭时，如果后台线程仍在运行，必须提示用户并安全停止。

## 影响范围

- `app/main_window.py`：后台 worker 信号、线程清理、按钮状态、打开/导出/关闭流程。
- `core/pipeline.py`：如有必要，补充处理完成状态所需的返回信息，但不改变检测、跟踪、计数算法语义。
- `analysis/heatmap.py` / GUI 调用侧：将查看热力图与保存热力图的交互语义明确拆分。
- 手工验证流程：覆盖正常处理、导出、取消、打开新视频、关闭窗口等用户路径。

## 非目标

- 不修改任务书、开题报告正文。
- 不改变 YOLOv11、Deep SORT、车辆 taxonomy、计数规则和分析结果包的数据语义。
- 不要求本变更把普通处理阶段同步保存完整处理后视频；如果实现选择临时视频缓存，需要保证不破坏现有导出行为。
