## 上下文

项目已经实现了基于 YOLOv11 + DeepSORT 的检测、跟踪、轨迹展示、车辆统计、热力图、训练和统一评估。但当前实现和论文文档之间存在几处关键偏差：

- 类别标签在源数据转换、训练配置、绘制层和统计层里各自维护，存在 `others` / `truck` / COCO 编号混用。
- 当前计数逻辑是 `track_id` 去重，而配置和论文里仍保留越线统计的语义。
- 热力图使用的是轨迹缓存的近期点，和“整段视频热力图”的描述不完全一致。
- 评估结果虽然存在，但覆盖序列太少，且缺少自采视频与实时性摘要。
- README 与代码框架不完全一致，`docs/` 目录也被整体忽略；任务书和开题报告为基线材料，只复核不直接修改。
- 视频处理链路缺少显式异常反馈，失败时用户体验和可解释性都偏弱。

## 目标 / 非目标

**目标：**

- 建立一个单一的 taxonomy 来源，让检测、统计、绘制、数据集转换、评估和文档使用同一套类别定义。
- 保留当前的累计计数能力，同时恢复可选的越线流量统计，并把流量窗口绑定到视频时间。
- 将轨迹展示缓存和热力图累计数据解耦，热力图反映完整运行过程，并支持保存到结果目录。
- 在热力图之外补充可直接用于论文展示的分析结果：车型结构、分时段流量、轨迹摘要、热点区域排名和处理性能摘要。
- 让评估输出更适合毕业设计展示：带时间戳、带环境信息、带 FPS/耗时、带公开序列与自采视频记录。
- 让 README 和仓库忽略规则与真实实现一致；任务书、开题报告作为基线文档保持原样。
- 让视频读取、导出和处理失败时有明确报错，不把问题隐藏在静默失败里。

**非目标：**

- 不重做 YOLOv11 或 DeepSORT 的核心算法结构。
- 不引入新的外部依赖，优先复用现有库和现有工程结构。
- 不做 Web 化或数据库化改造。
- 不扩展到跨摄像头协同或车牌识别等新的课题方向。

## 设计决策

### 决策 1: 使用单一的车辆 taxonomy 作为源头

将车辆类别定义抽象成单一来源，供 `config.yaml`、数据集 yaml、UI、统计和评估共享。实现时优先新增一个轻量共享模块，例如 `utils/taxonomy.py`，集中输出类别顺序、显示名称和反向映射，避免多个文件各写一套字典。

第四类保持为统一的“第四类/大车类”语义，并在文档中明确说明源数据集中的 `others`、`truck` 等标签会在转换阶段折算到同一类。这样可以保持跨数据集训练与 UI 显示的一致性。

### 决策 2: 把计数语义拆成两个模式

保留 `unique_track` 作为默认累计计数模式，用于回答“系统累计识别了多少辆车”。

新增或恢复 `line_crossing` 模式，用于回答“有多少车辆穿越了某条虚拟检测线”。该模式使用 `line_position` 和 `direction`，并把流量统计窗口绑定到视频时间，而不是处理线程的墙钟时间。

这样可以同时满足当前系统实现和论文中对交通流量分析的表达需要。

### 决策 3: 热力图与轨迹缓存分离

轨迹展示继续使用有限长度缓存，避免画面杂乱；热力图则维护独立的累积数据结构，按当前运行的完整帧序列生成。这样既保留实时展示，又让热力图具备“整段视频热点分布”的语义。

热力图需要支持保存两类结果：原始热力图和叠加到代表帧上的 overlay 图。保存路径放入本次处理或评估的结果目录，文件名带时间戳或视频名，避免覆盖。

### 决策 4: 分析结果围绕论文展示设计

除热力图外，优先补充不依赖额外标定信息的分析结果：

- 车型结构：累计数量、占比，可输出柱状图或饼图数据。
- 分时段流量：按视频时间分桶统计车辆数和分车型车辆数，可输出折线图数据。
- 轨迹摘要：每个 `track_id` 的类别、出现帧范围、持续时长、轨迹点数、像素路径长度。
- 热点区域排名：将画面划分为网格，根据热力图累计值输出 Top N 高密度区域。
- 处理性能摘要：视频帧数、处理耗时、平均 FPS、模型路径、设备信息。

不在无标定信息时输出真实车速。若后续需要速度估计，必须先增加像素到真实距离的标定配置。

### 决策 5: 评估输出以“结果包”而不是单个文本为中心

每次评估保留独立的时间戳目录，目录中同时包含：

- `summary.txt` 或等价摘要文件
- 检测和跟踪图表
- 运行时间、FPS、模型路径、序列信息
- 公开序列和自采视频的测试记录

如果图表生成失败，摘要文件仍必须写入，图表失败只降级为警告。

### 决策 6: 文档同步与仓库忽略规则同步推进

README 与真实代码保持同一套术语，尤其是 PySide6、类别命名、统计定义和评估覆盖范围。任务书 markdown、开题报告 markdown 作为基线材料仅进行复核，不在实现阶段改动正文。

`docs/` 目录不应继续被整目录忽略；如果需要保留旧版二进制文档，只应对旧格式做定向忽略，而不是把新的 markdown 文档一起挡掉。

### 决策 7: 处理失败要显式反馈

视频读取、写入和处理线程都要暴露清晰的错误路径：

- 打不开视频要提示用户
- 写入器初始化失败要提示用户
- 处理过程中异常要恢复按钮状态
- 模型切换与处理流程保持互斥

## 风险 / 权衡

- **第四类语义统一**：把不同来源的第四类折算到一个统一类会牺牲一点源标签语义，但能显著提高跨数据集训练、UI 展示和论文表述的一致性。
- **视频时间窗口依赖 FPS**：如果源视频 FPS 缺失或异常，需要采用可配置的兜底值，并在摘要中记录。
- **批量评估输出会增加运行时间**：这是可接受的，因为毕业设计更需要可追溯性，而不是只保留一次结果。
- **异常显式化可能暴露更多边界问题**：但这是必要的，静默失败更难在答辩中解释。
